# Разработка интерфейса сайта

## Подход: функциональность отдельно от дизайна

### Принцип разделения

1. **Функциональный слой** (что делает интерфейс)
   - Компоненты и их поведение
   - Логика работы с данными
   - Навигация и роутинг
   - Состояния и взаимодействия

2. **Визуальный слой** (как это выглядит)
   - Стили и токены
   - Анимации
   - Типографика
   - Цветовая схема

### Структура разработки

```
vovaipetrova-site/
├── src/
│   ├── core/              # Ядро функциональности (независимо от дизайна)
│   │   ├── components/    # Компоненты с логикой
│   │   ├── routing/       # Роутинг и навигация
│   │   ├── data/          # Работа с данными из core
│   │   └── state/         # Управление состоянием
│   ├── ui/                # Визуальный слой (можно менять отдельно)
│   │   ├── styles/        # CSS/стили
│   │   └── themes/        # Темы оформления
│   └── pages/             # Страницы (композиция компонентов)
└── docs/
    └── INTERFACE_SPEC.md  # Спецификация интерфейса
```

## Связь с ядром (core)

### Данные из core

Интерфейс получает данные через:
- `data/routes.json` — маршруты и метаданные
- `data/kb_glossary_lite.jsonl` — база знаний
- `data/stories_digests.jsonl` — Stories
- `data/tokens.json` — дизайн-токены (опционально)

### Контракт интерфейса

Интерфейс определяет:
1. **Какие данные нужны** (формат запроса)
2. **Как данные используются** (логика компонентов)
3. **Что отображается** (структура UI)

Ядро (core) предоставляет:
1. **Данные в нужном формате** (через content slices)
2. **Метаданные** (routes, OG tags)
3. **Токены** (для стилизации, опционально)

## Процесс разработки

### Этап 1: Спецификация функциональности
- Описать компоненты и их поведение
- Определить структуру данных
- Зафиксировать контракт с ядром

### Этап 2: Реализация логики
- Создать компоненты без стилей (только функционал)
- Реализовать роутинг
- Подключить данные из core

### Этап 3: Визуализация (позже)
- Применить стили/токены
- Добавить анимации
- Адаптивность

## Удержание контекста

### Документирование решений
- Все решения фиксируются в `docs/INTERFACE_SPEC.md`
- Компоненты документируются через JSDoc
- Контракты данных описываются в схемах

### Работа с контекстом
- Я могу удерживать контекст в рамках сессии
- Для больших проектов лучше документировать ключевые решения
- Используем структурированные файлы спецификаций
